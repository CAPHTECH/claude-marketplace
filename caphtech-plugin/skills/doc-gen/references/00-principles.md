# 原則: 生成AIによるドキュメント生成

## 能力モデル

### 強み（積極活用）

| 能力 | 理由 | 結果 |
|------|------|------|
| 大量情報の構造化要約 | 読む負荷を下げる | 理解の再現性が高まる |
| 散在断片の統合 | 全体像を作れる | オンボーディング・レビューの共通言語 |

### 弱み（対策必須）

| 傾向 | 理由 | 結果 |
|------|------|------|
| 根拠欠落箇所の補完 | 文章が整う | 誤りが見抜かれにくい（整っている嘘） |
| ハッピーパス偏重 | 成功パスが目立つ | 回帰・運用事故の地雷が残る |

→ スキルを「生成」ではなく **根拠・検証・更新** に寄せる

## 設計原則

### 1. 根拠リンク必須

重要な記述には必ずソースを紐づける:

```markdown
❌ 悪い例
UserServiceはユーザー認証を担当する。

✅ 良い例
UserServiceはユーザー認証を担当する。
(src/services/user.ts:1-150)
```

### 2. 事実と推測の分離

```markdown
❌ 悪い例
このキャッシュは性能最適化のために導入された。

✅ 良い例
このキャッシュが存在する。(src/cache/user.ts:45)
導入理由は不明（ADR未発見）。
```

### 3. 境界条件を優先

利用頻度ではなく以下で「代表」を選ぶ:

1. **変更頻度**: 頻繁に触る部分
2. **境界条件**: 認証/DB/非同期/外部連携/削除・退会
3. **不変条件**: 冪等性/整合性/権限/監査
4. **被害規模**: 障害時のインパクト
5. **追跡可能性**: デバッグの容易さ

### 4. 更新コストを考慮

- 少数のフロー（2-5本）でカバレッジ最大化
- 目的に直結しない詳細は省略
- 更新が必要な箇所を明示

## 危険領域

以下はAIが断定してはならない:

| 領域 | なぜ危険か | 扱い方 |
|------|-----------|--------|
| 意図・理由 | コードに書いていない | 草案として隔離、人間が決裁 |
| 性能・SLA | 実測が必要 | 「測定が必要」と明記 |
| 非同期・分散の細部 | 実行時に決まる | 代表失敗シナリオ必須 |
| セキュリティ保証 | 攻撃パターンは無限 | リスク示唆のみ、保証は不可 |

## 継続性のための設計

ドキュメントの価値 = **参照され続けること**

参照されなくなる原因:
1. 誤りがある → 根拠ゲートで対策
2. 古くなる → 更新コスト考慮した粒度
3. 実務で役立たない → タスクベース検証

→ 3パス読み取りはこれらを防ぐ構造
