# アートディレクション・ビジュアルコンポジション リファレンス（モバイル版）

モバイルアプリにおけるビジュアル構図、ブランド表現、コンテンツ戦略、モーション設計、画像ガイドラインの参照情報。

---

## 共通原則

### ワンコンポジション

アプリの各画面は**単一の構図**として設計する。カードモザイクやウィジェット風の寄せ集めにしない。

| ルール | 説明 |
|--------|------|
| ブランド優先 | アプリアイコン・ロゴ・ブランドカラーがアプリ全体で一貫して認識可能 |
| 表現力のあるタイポグラフィ | システムフォントのデフォルト設定に依存せず、ウェイト・サイズ・間隔で階層を表現 |
| セクション単一目的 | 各画面・各セクションは1つの目的に集中する |
| カード使用制限 | デフォルトはカードなし。相互作用が必要な場合のみカードを使用 |

### デザイン制約の事前定義

デザイン着手前に以下の制約を決定する:

| 制約 | 推奨上限 |
|------|----------|
| フォント種類 | 2種類まで（プラットフォームフォント + ブランドフォント） |
| アクセント色 | 1色 |
| プライマリCTA | 1画面に1個 |
| 表面階層 | 最大3層（background → surface → elevated surface） |

### コアデザイントークン

| トークン | 用途 |
|----------|------|
| background | 画面全体の背景 |
| surface | カード・シート等の表面 |
| primary text | 主要テキスト |
| muted text | 補助テキスト |
| accent | 強調・CTA |

## プロダクトUI設計（ミニマルアプローチ）

タスク指向のモバイルアプリは、情報密度と読みやすさの両立を目指す。

### 推奨パターン

| パターン | 説明 |
|----------|------|
| 落ち着いた表面階層 | background → surface → elevated surface の控えめな階層。過度なシャドウやボーダーで区切らない |
| 強いタイポグラフィ・間隔 | 装飾ではなくフォントサイズ・ウェイト・間隔で情報の優先順位を表現 |
| 少ない色 | アクセント色は1色、セマンティックカラーは必要最小限 |
| 濃密だが読みやすい情報 | モバイルの限られた画面を有効活用しつつ読みやすさを確保 |
| 最小限のクローム | 不要なボーダー・シャドウ・装飾・仕切り線を排除 |
| カードはインタラクション時のみ | 静的表示にカードを使わない。タップ・スワイプ等の操作がある場合のみ |

### 非推奨パターン

| パターン | 理由 |
|----------|------|
| カードモザイクが第一画面 | 情報の優先順位が不明確になる |
| すべての領域への太いボーダー/シャドウ | 視覚的ノイズが増え、プラットフォームのデザイン言語と乖離する |
| 装飾グラデーション背景 | ルーティンUIでは注意を散漫にする |
| 複数の競合するアクセント色 | 視覚的階層が崩れ、CTAが不明確になる |
| 意味のないアイコン | 認知負荷を増やす。ラベルだけで十分なら装飾アイコンは不要 |

## コピーガイドライン

### ユーティリティコピー

モバイルアプリのコピーは以下の優先順位で書く:

1. **方向付け**: ユーザーが今どこにいるか
2. **ステータス**: 現在の状態は何か
3. **アクション**: 次に何ができるか

### 避けるべき表現

- キャンペーン言語・比喩
- 長文の説明テキスト（1文で完結させる）
- 同じメッセージの異なる表現での繰り返し
- デザインコメンタリー（「美しい」「革新的」等の自己言及）

### コピーの品質チェック

- 見出し/ラベルだけスキャンして画面の目的が理解できるか？
- フィラーコピー（意味のない埋め合わせ文）がないか？
- 各セクション/画面に1つの責任しかないか？

## 画像ガイドライン

| 原則 | 説明 |
|------|------|
| 本物の画像を優先 | 実写・実製品の画像 ＞ 抽象イラスト・偽3D |
| テキスト上のコントラスト確保 | 画像上にテキストを置く場合、十分なコントラストを確保 |
| 埋め込みテキスト回避 | 画像内のテキストはローカライズ・アクセシビリティの障壁になる |
| 複数時点は個別画像 | コラージュではなく、スワイプやカルーセルで個別表示 |

### ビジュアルアンカー

装飾グラデーションではなく、実際の製品・環境・人物の画像を使用する:

- 削除するとアプリの意味が変わる（=不可欠である）
- コンテンツの文脈を伝えている
- テキストとの十分なコントラストを確保している

---

## モーション設計の制約原則

プラットフォーム固有のモーション仕様（Spring Animation / Material Motion）に加え、以下の制約原則を適用する。

### 意図的モーション原則

画面あたり**2-3の意図的なモーション**に限定する。

| チェック項目 | 基準 |
|-------------|------|
| 目的がある | 状態変化・フィードバック・ナビゲーションの意味がある |
| 速度 | 高速・控えめ。ユーザーを待たせない |
| 一貫性 | アプリ全体で一貫したモーション言語 |
| 必要性 | 装飾のみなら削除する |
| モバイル性能 | 低スペック端末でもスムーズに動作する |

## アンチパターン（避けるべき失敗）

| # | パターン | 問題 |
|---|---------|------|
| 1 | カードモザイクが第一画面 | ブランドの個性が伝わらない |
| 2 | 弱いブランド表現 | 何のアプリか一目で分からない |
| 3 | アクション不明確 | 次に何をすべきか分からない |
| 4 | テキスト背後の忙しい画像 | 可読性が損なわれる |
| 5 | 画面間で繰り返される気分説明 | 冗長で画面遷移のたびに同じメッセージ |
| 6 | 意味のないカルーセル/スワイプ | ユーザーが見逃す情報を隠す |
| 7 | 積み重ねカード構成 | カード濫用で情報階層が不明確 |

### 禁止事項

- デフォルトでカードを使わない
- 1画面に複数の支配的なアイデアを入れない
- フィラーコピーを入れない
- フォントを明確な理由なく2種類以上使わない
- 装飾目的のみのモーションを入れない

## 品質リトマステスト

デザイン完了時に以下を確認する。すべてYesであること:

| # | チェック項目 |
|---|-------------|
| 1 | アプリの目的が最初の画面で即座に理解できるか？ |
| 2 | 強いビジュアルアンカーまたは明確なCTAが1つ以上あるか？ |
| 3 | ラベル/見出しのスキャンのみで画面の役割が理解できるか？ |
| 4 | 各画面に1つの仕事しかないか？ |
| 5 | カードは本当に必要か？（なくても成立しないか確認） |
| 6 | モーションは状態変化や操作フィードバックの役割を果たしているか？ |
| 7 | 装飾シャドウ・ボーダーをすべて削除してもUI品質が保たれるか？ |
