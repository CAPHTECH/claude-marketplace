# AI 向け差分判定ルール

AI が作った成果物は、見た目の完成度に引きずられやすい。差分で見るときは次の既定ルールを使う。

## Rule A

コードが変わったのに、リンクされた要件、形式仕様、テストが一切変わっていないなら、まず不整合疑い。

必要な証拠:

- 変更した `C_i`
- 未変更の `REQ_i`, `F_i`, `T_i`

## Rule B

要件が変わったのに、形式仕様と property-based test が不変なら、まず形式化漏れ。

必要な証拠:

- 変更した `REQ_i`
- 未変更の `F_i`, `T_i`

## Rule C

テストがすべて成功していても、mutation survivor が多いなら、整合性保証とはみなさない。

必要な証拠:

- テスト結果
- mutation score または survivor 一覧

## Rule D

実行時 trace が要件IDを持たない機能は、要件適合を主張しない。

必要な証拠:

- `M_i` がない、または `req.id` が欠けた trace

## Rule E

独立に作った二つの形式化が一致しない要件は、要件そのものが曖昧。

必要な証拠:

- `F_i^A`
- `F_i^B`
- 相互含意の失敗か、分岐した制約

## 運用のコツ

- まず Rule A と Rule B を差分ゲートに置く
- Rule C は高リスク変更の昇格条件に使う
- Rule D は本番監視の最低条件にする
- Rule E は要件レビューに戻す条件にする
