## Design Thinking

人間中心のデザイン手法。共感・創造性・反復テストを通じて、ユーザーニーズに応えるソリューションを生成する。

### 5ステップ詳細

#### 1. Empathize（共感）

**目的**: ユーザーを観察・インタビューし、ニーズ・ペインポイントを理解する。

**実施内容**:
- ユーザーインタビュー（5人以上で主要パターンを発見）
- 行動観察（実際のタスク遂行を観察）
- ペインポイントの洗い出し
- 共感マップの作成（Says / Thinks / Does / Feels）

**Claudeでの適用**:
- ユーザーに現在の課題・不満を聞き出す
- 既存ソリューションの不満点を整理する
- ペルソナを共同で定義する

#### 2. Define（定義）

**目的**: 問題を明確に定義し、解決すべき課題を言語化する。

**実施内容**:
- 課題ステートメントの作成: 「[ユーザー]は[ニーズ]を必要としている。なぜなら[インサイト]だから」
- How Might We (HMW) 質問の生成
- 優先順位の決定

**Claudeでの適用**:
- ユーザーの回答からHMW質問を3〜5個生成する
- ユーザーと優先順位を決める

#### 3. Ideate（発想）

**目的**: 多数のアイデアを生成し、解決策の幅を広げる。

**実施内容**:
- ブレインストーミング（量を重視）
- Crazy 8s（8分で8つのスケッチ）
- 類似プロダクトからのインスピレーション

**Claudeでの適用**:
- 3〜5つの異なるアプローチを提案する
- 各アプローチのPros/Consを提示する
- ユーザーに選択してもらう

#### 4. Prototype（試作）

**目的**: 低忠実度プロトタイプを素早く作成し、アイデアを形にする。

**実施内容**:
- ワイヤーフレーム作成
- 主要フローの可視化
- インタラクションの概要設計

**Claudeでの適用**:
- ASCIIアートで画面レイアウトを提示する
- 画面遷移フロー図を作成する
- HTML/CSSでプロトタイプコードを生成する

#### 5. Test（検証）

**目的**: プロトタイプをユーザーに触ってもらいフィードバックを収集する。

**実施内容**:
- ユーザビリティテスト（5人で問題の85%を発見）
- フィードバックの収集と整理
- 改善点の優先順位付け

**Claudeでの適用**:
- デザイン案に対するユーザーのフィードバックを整理する
- ヒューリスティック評価で潜在的な問題を指摘する
- 改善案を提示する

## Design Sprint（Google Ventures式）

5日間でアイデアからプロトタイプ・検証まで完了する集中型プロセス。

### 5日間スケジュール

| 日 | フェーズ | 内容 | アウトプット |
|----|---------|------|-------------|
| Day 1 | Map（マッピング） | ゴール設定、課題の可視化、ターゲットユーザー選定 | 課題マップ、スプリントゴール |
| Day 2 | Sketch（スケッチ） | 個人でソリューションをスケッチ、参考事例の収集 | ソリューションスケッチ群 |
| Day 3 | Decide（決定） | 投票でベストアイデアを選定、ストーリーボード作成 | 決定案、ストーリーボード |
| Day 4 | Prototype（プロトタイプ） | リアルなプロトタイプを1日で作成 | テスト可能なプロトタイプ |
| Day 5 | Test（テスト） | 5人のユーザーにテスト、パターンを分析 | テスト結果、次のアクション |

### Claudeでの適用

Design Sprintの各日の活動を1セッション内で凝縮して実施する:
1. 課題の整理とゴール設定
2. 複数のソリューション案を提示
3. ユーザーと最適案を選定
4. HTML/CSSプロトタイプの生成
5. ヒューリスティック評価による検証

## Double Diamond

英国Design Councilが2003年に提唱。発散→収束を2回繰り返すフレームワーク。

### プロセス図

```
        Diamond 1              Diamond 2
  (正しい問題を見つける)    (正しい解決策を作る)

      ╱  ╲                    ╱  ╲
     ╱    ╲                  ╱    ╲
    ╱      ╲                ╱      ╲
   ╱        ╲              ╱        ╲
  Discover  Define       Develop   Deliver
  (発散)    (収束)       (発散)    (収束)
```

### 4フェーズ詳細

| フェーズ | 方向 | 目的 | 主要活動 |
|----------|------|------|----------|
| Discover | 発散 | 広くリサーチし、課題・文脈を理解 | ユーザーリサーチ、市場調査、競合分析 |
| Define | 収束 | インサイトを絞り込み、問題を明確化 | 課題ステートメント、ペルソナ定義、要件定義 |
| Develop | 発散 | 多数のアイデアを出し、プロトタイプで実験 | ブレインストーミング、ワイヤーフレーム、プロトタイプ |
| Deliver | 収束 | テスト・改善を繰り返し、最終ソリューションを完成 | ユーザーテスト、A/Bテスト、最終デザイン |

## Atomic Design

Brad Frost（2013年）が提唱。UIを5階層で構成する方法論。

### 5階層詳細

#### 1. Atoms（原子）

最小のUI要素。これ以上分解できない。

| カテゴリ | 例 |
|----------|-----|
| インタラクティブ | Button, Input, Checkbox, Radio, Toggle, Select |
| テキスト | Label, Heading, Paragraph, Link, Badge |
| ビジュアル | Icon, Avatar, Divider, Spinner, Skeleton |
| フォーム | InputField, TextArea, DatePicker |

**設計時の注意点**:
- 全状態（Idle, Hover, Active, Focused, Disabled, Loading）を定義する
- サイズバリアント（sm, md, lg）を用意する
- アクセシビリティ（キーボード操作、ARIA属性）を組み込む

#### 2. Molecules（分子）

Atomsの組み合わせで機能を持つ小単位。

| 例 | 構成Atoms |
|-----|----------|
| SearchBar | Input + Button + Icon |
| FormField | Label + Input + HelperText |
| NavItem | Icon + Label + Badge |
| MenuItem | Icon + Label + Shortcut |
| Breadcrumb | Link + Divider + Link |

**設計原則**: 単一の責務を持つ。1つのMoleculeは1つの機能を果たす。

#### 3. Organisms（有機体）

Moleculesの組み合わせで独立したUIセクション。

| 例 | 構成要素 |
|-----|---------|
| Header | Logo + Navigation + SearchBar + UserMenu |
| Sidebar | NavItemのリスト + SectionDivider |
| DataTable | TableHeader + TableRow + Pagination |
| CardList | Cardのグリッド + FilterBar |
| Footer | NavLinks + CopyrightText + SocialLinks |

**設計原則**: 独立して再利用可能。異なるページに配置しても機能する。

#### 4. Templates（テンプレート）

Organismsをページ構造に配置したワイヤーフレーム。

| 例 | 構造 |
|-----|------|
| DashboardLayout | Header + Sidebar + MainArea + Footer |
| AuthLayout | CenteredCard + BrandingArea |
| SettingsLayout | Header + SettingsNav + ContentArea |
| ListDetailLayout | Header + ListPanel + DetailPanel |

**設計原則**: コンテンツに依存しない構造の定義。プレースホルダで構成を示す。

#### 5. Pages（ページ）

Templateに実際のコンテンツを流し込んだ最終形。

**設計原則**: 実データでTemplateの有効性を検証する。エッジケース（長いテキスト、空データ、大量データ）も考慮する。

### ディレクトリ構造の例

```
components/
  atoms/       → Button, Input, Icon, Badge, Label
  molecules/   → SearchBar, FormField, NavItem, MenuItem
  organisms/   → Header, Sidebar, DataTable, CardList
  templates/   → DashboardLayout, AuthLayout, SettingsLayout
  pages/       → DashboardPage, SettingsPage, LoginPage
```

## Nielsenの10ユーザビリティヒューリスティクス

Jakob Nielsen（1994年）が導出した10原則。デザインレビュー・品質確認のチェックリストとして使用する。

### 一覧

| # | 原則 | 説明 | チェック観点 |
|---|------|------|-------------|
| 1 | システム状態の可視化 | 現在何が起きているかをフィードバック | ローディング表示、進捗バー、成功/エラー通知 |
| 2 | 現実世界との一致 | ユーザーの言葉・概念を使用 | 専門用語を避ける、メタファーの適切さ |
| 3 | ユーザーのコントロールと自由 | 取り消し・やり直しを常に可能に | Undo/Redo、モーダルの閉じ方、戻るボタン |
| 4 | 一貫性と標準 | 同じ操作は同じ結果を生む | 用語・アイコン・レイアウトの統一 |
| 5 | エラー防止 | エラーが起こりにくい設計 | 確認ダイアログ、入力制約、デフォルト値 |
| 6 | 再認 vs 想起 | 情報を見せて記憶負荷を減らす | 最近使ったもの、サジェスト、ツールチップ |
| 7 | 柔軟性と効率 | 初心者にも熟練者にも対応 | ショートカット、カスタマイズ、クイックアクション |
| 8 | 美的でミニマルなデザイン | 不要な情報を排除 | 視覚的ノイズの削減、情報の優先順位 |
| 9 | エラーからの回復支援 | エラーメッセージを具体的に | 原因の提示、解決策の提案、やり直しの容易さ |
| 10 | ヘルプとドキュメント | 必要な時にアクセスできるヘルプ | オンボーディング、ツールチップ、ヘルプセンター |

### 評価手順

1. 各原則について対象UIを確認する
2. 問題を発見した場合、深刻度を0〜4で評価する:
   - 0: ユーザビリティ問題ではない
   - 1: 外見上の問題（修正は低優先度）
   - 2: 軽微な問題（修正は中優先度）
   - 3: 重大な問題（修正は高優先度）
   - 4: 致命的な問題（リリース前に修正必須）
3. 問題の一覧と改善案をまとめる

## ユーザーテスト手法

### シンクアラウド（Think Aloud Protocol）

ユーザーが操作しながら思考を声に出す手法。5人のユーザーで問題の85%を発見可能。

**手順**:
1. テストシナリオ（タスク）を準備する
2. ユーザーに操作しながら考えていることを話してもらう
3. 観察者は介入せず記録する
4. 終了後にフォローアップ質問を行う

### A/Bテスト

2バージョンを同時提供し、定量的に比較する手法。

**評価指標の例**:
- コンバージョン率
- タスク完了率
- タスク完了時間
- エラー率
- ユーザー満足度スコア

### カードソーティング

コンテンツカードをユーザーにグルーピングさせ、情報アーキテクチャを設計する手法。

- **オープンソート**: ユーザーが自由にグループと名前を決める
- **クローズドソート**: 事前に決めたカテゴリにカードを分類する

### ツリーテスト

テキストのみのサイト構造でユーザーに目的の情報を探させ、ナビゲーション構造を検証する手法。

### ヒューリスティック評価

専門家がNielsenの10原則等でUIを評価する手法。低コスト・短時間で実施可能。ユーザーテストとの併用が最も効果的。
