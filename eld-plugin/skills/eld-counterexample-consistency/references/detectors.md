# 検出器カタログ

このスキルの中心は、文章の感想ではなく、反例を返せる検出器を持つことにある。

## 6つの検出器

| 検出器 | 問うこと | 最低限の出力 |
|--------|----------|--------------|
| 欠落検出器 | 要件ごとに必要なノードと辺がそろっているか | 欠けている成果物またはリンク |
| 二重形式化差分検出器 | 独立に作った二つの形式化が同じ意味か | 相互含意の失敗箇所、または分岐した制約 |
| 仕様内部矛盾検出器 | 制約集合が成立可能か | unsat、vacuity、到達不能、時間衝突 |
| テスト妥当性検出器 | テストが要件を本当に担保しているか | 最小反例、mutation survivor |
| 契約違反検出器 | コードの局所意味が守られているか | precondition、postcondition、invariant の違反 |
| 実行時観測整合性検出器 | 実運用の振る舞いが要件に一致するか | trace rule 違反、span 属性の欠落 |

## 1. 欠落検出器

最初に見るのは矛盾ではなく欠落。

- 要件に形式仕様がない
- 要件にテストがない
- 要件に観測点がない
- テストがどの要件を担保するか不明
- コード変更があったのに、リンクされた要件やテストが更新されていない

## 2. 二重形式化差分検出器

同じ要件から、独立した二系統の形式化を作る。

- 別モデル
- 別プロンプト
- 別担当者
- 別表現系

そのうえで `F_i^A ⇒ F_i^B` と `F_i^B ⇒ F_i^A` を確認する。片方でも崩れたら、要件の曖昧さか形式化の分岐を疑う。

## 3. 仕様内部矛盾検出器

見る対象は次の4つ。

- 充足不能
- 空虚真
- 到達不能状態
- 時間制約衝突

反例が出ない場合も、空虚真や unsat core のような局所化情報を証拠として保存する。

## 4. テスト妥当性検出器

例示テストの有無だけで終わらせない。

- property-based test で境界と一般性を見る
- stateful test で操作列依存の破綻を見る
- mutation testing で鋭さを測る

mutation survivor は「テストがあっても保証が弱い」ことを示す証拠として扱う。

## 5. 契約違反検出器

テストより細かい粒度で止める。

- precondition
- postcondition
- invariant

局所契約で止まるなら、要件違反をより早い位置で検出できる。

## 6. 実行時観測整合性検出器

trace または event に最低でも次を持たせる。

- `req.id`
- `state.before`
- `state.after`
- `decision`
- `external_call`
- `timeout_ms`

これで「A の後に B が 5 秒以内に起きる」「C は起きてはいけない」のような監視則が書ける。

## 共通通貨

検出器の出力は、次のどれかに還元する。

- 反例
- 変異生存
- 契約違反
- トレース違反
- 欠落

どれにも落ちない指摘は、参考情報として別枠に下げる。
