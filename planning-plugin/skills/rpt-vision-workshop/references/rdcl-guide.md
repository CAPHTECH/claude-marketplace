# RDCL Strategy Guide

RDCLストラテジーの各要素の定義、具体例、深掘り方法。

## 概要

RDCLはビジョンを実現するための戦略を4要素で構造化するフレームワーク。**順番が重要** — Real Painから始め、Logisticsで終わる。ビジネスモデルから逆算するのではなく、ユーザーの痛みから始める。

## Real Pain（真の痛み）

### 定義
ユーザーが実際に感じている、解決に値する痛み。推測ではなく検証された痛み。

### 検証方法
- **頻度**: どのくらいの頻度で発生するか？（毎日 > 週1 > 月1）
- **深刻度**: 不便 < 困る < 業務に支障 < 致命的
- **代替手段**: 現在どう対処しているか？そのコストは？
- **支払意思**: この問題の解決にお金を払うか？

### 良いReal Painの例
- 「中小企業の経理担当者は月末に3日間かけて手作業で請求書を突合している」
- 「リモートチームのマネージャーはメンバーの作業状況を把握するために1日3回Slackを巡回している」

### 悪いReal Painの例（避けるべき）
- 「人々はもっと便利なツールを求めている」（曖昧）
- 「市場にはまだこの機能を持つ製品がない」（痛みではなく機会）

## Design（設計）

### 定義
Real Painを解消するプロダクトの設計思想。何をするかだけでなく、**何をしないか**も含む。

### トレードオフの明確化
- シンプルさ vs 機能の豊富さ
- カスタマイズ性 vs 使いやすさ
- 自動化 vs ユーザーコントロール
- スピード vs 精度

### 設計決定の記述方法
```
我々は [設計判断A] を選択する。
なぜなら [根拠] だからである。
これにより [トレードオフで失うもの] は犠牲になるが、
[得られる価値] が対象ユーザーにとって重要だからである。
```

### 例
「我々はAI自動分類を選択する。手動タグ付けの正確性は犠牲になるが、月末3日の作業を30分に短縮できるため、経理担当者にとってはその方が重要。」

## Capabilities（能力）

### 定義
設計を実現するために必要な能力・技術・リソース。現状とのギャップ分析が核心。

### ギャップ分析の観点
- **技術**: 必要な技術スタックと現在の技術力
- **人材**: 必要なスキルセットと現在のチーム構成
- **データ**: 必要なデータと現在のアクセス状況
- **パートナーシップ**: 外部連携の必要性

### 記述方法
| 必要な能力 | 現状 | ギャップ | 対策 | 期間 |
|-----------|------|---------|------|------|
| (例) ML推論基盤 | プロトタイプのみ | 本番運用レベル未達 | クラウドMLサービス活用 | 2ヶ月 |

## Logistics（物流・ビジネスモデル）

### 定義
プロダクトをユーザーに届け、持続的に運営するための仕組み。

### 検討項目
- **チャネル**: どうやってユーザーに届けるか
- **収益モデル**: サブスクリプション / 従量課金 / フリーミアム / マーケットプレイス
- **価格設定**: ユーザーの支払意思に基づく価格帯
- **顧客獲得**: 最初の100人をどう獲得するか
- **顧客維持**: チャーンを防ぐ仕組み

### ビジネスモデル検討の注意点
- ビジョンとビジネスモデルの整合性を確認
- 短期的な収益化がビジョンを損なわないか
- ユーザーの痛みの深刻度と価格が見合っているか

## RDCL間の整合性チェック

各要素が矛盾なく繋がっているか確認する：

```
Real Pain → Design: この痛みをこの設計で本当に解消できるか？
Design → Capabilities: この設計を実現する能力があるか（または獲得できるか）？
Capabilities → Logistics: この能力で持続的に届けられるか？
Logistics → Real Pain: このビジネスモデルでユーザーの痛みに集中し続けられるか？
```
