# Documentation

Use this route for onboarding docs, architecture notes, spec extraction, or change guides.

## Plan Priorities

- identify the audience
- define the source artifacts to read
- keep claims traceable to code or evidence
- structure the output around how the reader will use it

## Include

- audience and use case
- artifacts to inspect
- sections to produce
- traceability or citation expectations
- gaps or uncertainties to call out

## Avoid

- rewriting implementation details without a reader need
- mixing as-is and to-be statements without labeling them
- overexplaining generic concepts the audience already knows

## Good Shape

The plan should make clear:

- what is being documented
- who it is for
- what evidence backs each major claim
