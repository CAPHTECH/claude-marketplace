# CLAUDE.md生成ガイド

回答パターンに基づいて、適切なセクションを組み合わせてCLAUDE.mdを生成する。

## 基本構造

```markdown
# CLAUDE.md - AI開発アシスタント設定

## 原則
[基本方針]

## コミュニケーション
[言語・応答スタイル]

## 開発スタイル
[開発アプローチ]

## AI協働
[AIの役割・介入度]

## 技術スタック
[言語・フレームワーク情報]
```

## セクション別パターン

### 原則セクション

**共通項目**（常に含める）:
```markdown
## 原則
- ライブラリを切り替える場合は必ずユーザーに許可を得ること
```

### コミュニケーションセクション

**言語設定パターン**:

| 回答 | 生成内容 |
|------|----------|
| 全て日本語 | `- 応答言語: 日本語`<br>`- コード内コメント: 日本語`<br>`- 変数名・関数名: 英語（コメントは日本語）` |
| 応答は日本語、コードは英語 | `- 応答言語: 日本語`<br>`- コード内コメント: 英語`<br>`- 変数名・関数名: 英語` |
| 全て英語 | `- 応答言語: English`<br>`- コード内コメント: English` |

**詳しさパターン**:

| 回答 | 生成内容 |
|------|----------|
| 簡潔 | `- 説明は簡潔に。詳細が必要な場合はユーザーが質問する` |
| 詳しく | `- 実装の背景、トレードオフ、代替案を含めて詳しく説明` |
| 状況次第 | `- 複雑な処理は詳しく、単純な処理は簡潔に` |

**提案スタイルパターン**:

| 回答 | 生成内容 |
|------|----------|
| 複数提示 | `- 選択肢がある場合は2-3個の案を提示し、各案のメリット・デメリットを説明` |
| ベスト1つ | `- 最適と判断した案を1つ提示（理由付き）。ユーザーが求めた場合のみ代替案を提示` |
| 重要な分岐点のみ複数 | `- 重要なアーキテクチャ判断のみ複数案を提示。通常の実装はベスト案のみ` |

### 開発スタイルセクション

**アプローチパターン**:

| 回答 | 生成内容 |
|------|----------|
| 動くもの優先 | `- まず動くコードを作成し、後でリファクタリング`<br>`- 最初から完璧を目指さない` |
| 設計優先 | `- 実装前に設計を検討`<br>`- インターフェースを先に定義してから実装` |
| ケースバイケース | `- 小さな機能は動くもの優先、大きな機能は設計優先` |

**技術選択パターン**:

| 回答 | 生成内容 |
|------|----------|
| 新しい技術を積極的に | `- 新しいライブラリやパターンを積極的に提案`<br>`- 実験的な機能も選択肢に含める` |
| 安定重視 | `- 実績のある安定したライブラリを優先`<br>`- 新しい技術を提案する際は明示的に確認` |
| バランス | `- 安定性を基本としつつ、明確なメリットがあれば新技術を提案` |

**テストパターン**:

| 回答 | 生成内容 |
|------|----------|
| TDD | `- TDD（テスト駆動開発）で実装`<br>`- テストを先に書いてから実装` |
| 後からテスト | `- 実装完了後にテストを追加`<br>`- 重要なロジックを中心にテスト` |
| 最小限 | `- テストはユーザーの指示があった場合のみ作成` |

**コメントパターン**:

| 回答 | 生成内容 |
|------|----------|
| 多め | `- コードには詳細なコメントを記述`<br>`- 関数の目的、引数、戻り値を必ず説明` |
| 最小限 | `- コメントは最小限に抑え、コード自体を読みやすく` |
| 重要箇所のみ | `- 複雑なロジック、重要な設計判断にのみコメント`<br>`- 「なぜ」そうしたかを説明` |

### AI協働セクション

**役割パターン**:

| 回答 | 生成内容 |
|------|----------|
| コード生成まで全部 | `- 主な役割: 完成コードの生成`<br>`- 実行可能なコードを提供` |
| レビューと提案 | `- 主な役割: コードレビュー、改善提案`<br>`- ユーザーのコードを分析し、問題点と改善案を提示` |
| 設計相談・壁打ち | `- 主な役割: 設計相談、アーキテクチャ検討`<br>`- 実装方針の議論、トレードオフの分析` |
| ヒントだけ | `- 主な役割: ヒント提供、方向性の提示`<br>`- 完成コードは提供せず、考え方や参考資料を提示` |

**出力形式パターン**:

| 回答 | 生成内容 |
|------|----------|
| 完成コード | `- 実行可能な完成コードを提供`<br>`- 説明は最小限に` |
| アプローチ提示 | `- 実装アプローチを先に提示`<br>`- ユーザーの確認後にコードを生成` |
| 両方 | `- アプローチを説明しながらコードを提示` |

**介入度パターン**:

| 回答 | 生成内容 |
|------|----------|
| 積極的に | `- コードの問題点や改善点を積極的に指摘`<br>`- セキュリティ、パフォーマンス、可読性の観点からレビュー` |
| 聞かれたら | `- レビューはユーザーの依頼があった場合のみ`<br>`- 依頼された範囲に限定して回答` |
| 重大な問題のみ | `- セキュリティリスクや重大なバグのみ指摘`<br>`- スタイルや軽微な改善は指摘しない` |

### 技術スタックセクション

ユーザーの回答をそのまま反映:

```markdown
## 技術スタック
- 主要言語: [Q1.1の回答]
- フレームワーク: [Q1.1の回答から抽出]
- 得意領域: [Q1.3の回答]
- 苦手領域: [Q1.3の回答] → この領域は詳しく説明
```

## 生成例

### パターン1: 経験豊富なバックエンドエンジニア

回答:
- 技術: Python, FastAPI
- スタイル: 設計優先、TDD
- AI役割: コード生成
- コミュニケーション: 簡潔、日本語

```markdown
# CLAUDE.md - AI開発アシスタント設定

## 原則
- ライブラリを切り替える場合は必ずユーザーに許可を得ること

## コミュニケーション
- 応答言語: 日本語
- 説明は簡潔に。詳細が必要な場合はユーザーが質問する
- 最適と判断した案を1つ提示（理由付き）

## 開発スタイル
- 実装前に設計を検討
- TDD（テスト駆動開発）で実装
- 複雑なロジック、重要な設計判断にのみコメント

## AI協働
- 主な役割: 完成コードの生成
- コードの問題点や改善点を積極的に指摘

## 技術スタック
- 主要言語: Python
- フレームワーク: FastAPI
```
