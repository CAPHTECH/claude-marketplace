# Description 記述ガイド

descriptionはClaudeがスキルを選択する唯一の判断材料。
100以上のスキルから正しく選ばれるために、明確で具体的に書く。

## 必須要素

1. **何をするか** — スキルの機能を具体的に
2. **いつ使うか** — トリガーとなる状況・フレーズ

## フォーマット

- 単一行（YAMLマルチライン `|` や `>` は使わない）
- 1024文字以内
- XMLタグ（`<`, `>`）不可
- 三人称で記述

## テンプレート

### 基本テンプレート

```
[機能の説明]。[トリガー条件の列挙]。
```

### 詳細テンプレート（推奨）

```
[動詞で始まる機能説明]。[追加機能があれば列挙]。Use when [条件1], [条件2], or [条件3]。
```

## Good Examples

### ツール統合型

```yaml
description: PDFファイルからテキスト・テーブルを抽出し、フォーム入力・ドキュメント結合を行う。PDFの操作、フォーム入力、ドキュメント抽出に関する依頼時に使用する。
```

分析:
- 具体的な機能を列挙（抽出、フォーム入力、結合）
- トリガーキーワードが明確（PDF、フォーム、抽出）

### ワークフロー型

```yaml
description: Git差分を分析して記述的なコミットメッセージを生成する。コミットメッセージの作成やステージされた変更のレビュー時に使用する。
```

分析:
- アクション（分析→生成）が明確
- 具体的なトリガー（コミットメッセージ、ステージされた変更）

### ドメイン知識型

```yaml
description: このコードベースのAPI設計パターンとエラーハンドリング規約。APIエンドポイントの追加・修正・レビュー時に自動参照される。
```

分析:
- スコープが明確（API設計パターン、エラーハンドリング）
- 自動参照のトリガー条件が具体的

## Bad Examples

### 曖昧すぎる

```yaml
# NG
description: ドキュメントを扱う
description: データを処理する
description: ファイル操作のヘルパー
```

問題: Claudeがいつ使うべきか判断できない

### 機能しか書いていない

```yaml
# NG
description: TypeScriptコードの静的解析を行う
```

問題: 「いつ使うか」がない。以下のように修正:

```yaml
# OK
description: TypeScriptコードの静的解析を行い、型安全性・パフォーマンス・セキュリティの問題を検出する。コードレビュー時、PR作成前、「型チェックして」「静的解析して」と言われた時に使用する。
```

### トリガーが本体にある

```yaml
# NG: descriptionにトリガーがなく、本体に「When to Use」セクション
description: コードの品質分析ツール
---
## When to Use This Skill
- コードレビュー時
- PR作成前
```

問題: 本体はスキル起動後にしか読まれない。トリガー情報はdescriptionに書く。

## キーワード選定のコツ

- ユーザーが自然に使う言葉を含める
- 日本語と英語の両方のキーワードを入れる（バイリンガル環境向け）
- 類義語を網羅する（「レビュー」「チェック」「確認」等）
- 具体的なファイル拡張子やツール名を入れる（`.pdf`, `Docker`, `pytest`等）

## チェックリスト

- [ ] 何をするか明記されている
- [ ] いつ使うか（トリガー条件）が含まれている
- [ ] 単一行で記述されている
- [ ] 1024文字以内
- [ ] XMLタグを含まない
- [ ] ユーザーが使うキーワードが含まれている
- [ ] 曖昧な表現がない
