# Schema Balance

AI支援作業では、上位の分解・比較・評価スキーマは強まりやすい一方で、下位の局所因果・成立条件・障害モードのスキーマは痩せやすい。
このスキルは、作業中に両者を意図的に同時運用し、説明だけ立派で壊れやすい変更を減らす。

## 2つの対象

### 強化する側

- 問題分解
- 要件から設計候補への写像
- 批判的レビュー
- 仕様化と制約明示
- テスト観点の設定
- ドメイン理解
- 委譲と統制

### 維持する側

- 局所因果
- 状態遷移
- 隠れた依存
- 境界条件
- 障害モード
- 並行性、ライフサイクル、性能の罠
- パターンの適用条件と不適用条件

## ワークフロー

### Step 1: スキーマリスクを先に選ぶ

作業開始時に、以下をそれぞれ最大2つずつ選ぶ。

- `strengthen_now`: 今回の作業で強化したい上位スキーマ
- `preserve_now`: 今回の作業で痩せさせたくない下位スキーマ

選び方:

- タスクが曖昧なら `問題分解` または `仕様化`
- 案が複数あるなら `設計候補の比較`
- AIのコードを多く採用するなら `批判的レビュー`
- 差分が広いなら `局所因果` と `隠れた依存`
- 状態を持つ処理なら `状態遷移`
- 非同期や購読があるなら `並行性/ライフサイクル`
- 例外系が重要なら `境界条件` と `障害モード`

### Step 2: 2系統ブリーフを作る

Step 2で [assets/schema-balance-brief.md](assets/schema-balance-brief.md) を使い、上位と下位を1枚に置く。

最低限、以下を埋める。

- `goal`
- `strengthen_now`
- `preserve_now`
- `touched_path`
- `success_path`
- `failure_path`
- `hidden_dependency`
- `human_owned_slice`
- `first_validation`

ここで重要なのは、「何を考えるか」だけでなく「何を自分が理解責任として持つか」を固定すること。

### Step 3: 生成前に上位スキーマを1回使う

コード生成や実装前に、次のどれかを必ず1つ行う。

- 問題を3〜6個の単位に分解する
- 設計候補を2案以上並べて、採用理由と不採用理由を1行ずつ書く
- 固定条件、可変条件、禁止事項を先に出す
- 正常系、否定例、境界例を1つずつ決める

AIに依頼する前に、この上位判断を人間側で明示する。

### Step 4: 受け入れ前に下位スキーマを1回使う

AIの出力を採用する前に、次を必ず行う。

- 変更が成立する理由を3〜5行で説明する
- 重要な `success_path` を1本トレースする
- 重要な `failure_path` を1本トレースする
- 触る状態、依存、境界のどれか1つ以上を明示する

説明できない場合は、差分を縮めるか、関連コードを読んでから採用する。

### Step 5: 人間の担当領域を残す

毎タスクで、少なくとも1つは人間が自分で持つ。

候補:

- テストケースの追加または修正
- 失敗モードの列挙
- 状態遷移の確認
- 変更波及の追跡
- 境界条件の洗い出し
- 監視、ログ、メトリクスの確認

`human_owned_slice` は委譲しすぎを防ぐための固定点として使う。

### Step 6: 検証と学習回収

変更後は、上位と下位の両方を1行ずつ回収する。

- `strengthened`: 今回、何の上位スキーマが少し良くなったか
- `preserved`: 今回、どの下位スキーマを痩せさせずに済んだか
- `remaining_blind_spots`: まだ読めていない因果、境界、障害モード

検証は、`preserve_now` に対応するものを優先する。

例:

- `境界条件` を選んだなら否定例または境界例を回す
- `局所因果` を選んだなら変更波及を説明する
- `状態遷移` を選んだなら遷移前後の不変条件を見る
- `並行性` を選んだならタイミング依存の崩れ方を確認する

## 停止条件

以下のどれかに当たったら、そのまま生成を続けない。

- 何が状態として変わるのか説明できない
- 成功経路は言えるが失敗経路が言えない
- 複数モジュールにまたがるのに因果の説明がない
- 設計案を選んだ理由が「AIがそう書いたから」しかない
- テストが通っても、何を守ったのか自分で言えない

## 連携

主な詰まりに応じて、以下へ送る。

| 詰まり | 接続先 |
|---|---|
| 仕様や制約が曖昧 | `spec-observation`, `uncertainty-resolution` |
| 変更波及が読めない | `impact-analysis`, `ai-led-onboarding` |
| 境界条件が弱い | `boundary-observation`, `systematic-test-design` |
| 並行性、タイミング依存が不安 | `concurrency-observation` |
| 監視や復旧が弱い | `operability-observation` |
| 最終レビューを厳しくしたい | `critical-code-review` |

## ガードレール

1. 上位スキーマだけで安心しない
2. 下位スキーマだけで局所最適に閉じない
3. 自分で説明できない差分を、そのまま責任範囲に入れない
4. 差分が大きい時ほど `human_owned_slice` を増やす
5. 最低1つは失敗側の観点で検証する

## 出力

- `schema_balance_brief`
- `strengthened`
- `preserved`
- `remaining_blind_spots`
- `recommended_followup_skills`
