# Bottleneck Catalog

Step 2 の 5領域スキャンで読む。各領域について、詰まりのシグナル、今すぐ解消できる内容、解消不能時の接続先を素早く決めるための表。

## 1. 要件

| 項目 | 内容 |
|---|---|
| 判定シグナル | 成功条件が曖昧、用語の解釈が揺れる、否定例がない、完了条件がない |
| 根拠として拾うもの | Issue本文、仕様書、テスト名、既存エラーメッセージ |
| 今すぐ解消すること | 仮定を2〜5件で明文化、成功条件を最大3項目に圧縮、否定例を1つ置く |
| いま解消しないこと | 仕様交渉、要件の正式決裁、大きなスコープ再設計 |
| エスカレーション先 | `spec-observation`, `uncertainty-resolution` |

## 2. コンテキスト

| 項目 | 内容 |
|---|---|
| 判定シグナル | 入口が分からない、読む順番が曖昧、source of truth が複数候補、関連コードが広すぎる |
| 根拠として拾うもの | エントリポイント、関連テスト、ADR、直近変更履歴 |
| 今すぐ解消すること | source of truth を primary/secondary に分ける、読む順番を3〜5手に固定する |
| いま解消しないこと | コードベース全体の網羅調査、設計史の完全復元 |
| エスカレーション先 | `impact-analysis`, `ai-readability-analysis` |

## 3. 境界

| 項目 | 内容 |
|---|---|
| 判定シグナル | どこまで触るか不明、副作用や外部依存が見えない、所有責務が曖昧 |
| 根拠として拾うもの | モジュール境界、I/O、外部API、DB、設定値、関連ジョブ |
| 今すぐ解消すること | `in/out/depends/owns/avoid_touching` を埋める、触る境界と触らない境界を分ける |
| いま解消しないこと | 境界再設計、責務再編、大規模分割 |
| エスカレーション先 | `impact-analysis`, `dependency-observation`, `security-observation`, `concurrency-observation` |

## 4. 検証

| 項目 | 内容 |
|---|---|
| 判定シグナル | 最初に回すテストがない、成功シグナルが曖昧、境界ケースが漏れている |
| 根拠として拾うもの | 既存テスト、CI設定、失敗ログ、再現手順 |
| 今すぐ解消すること | 最初に回す1つのテストまたは確認手順を決める、止める条件を1つ決める |
| いま解消しないこと | 新規テスト群の大量実装、PBT導入、包括的QA計画 |
| エスカレーション先 | `observation-minimum-set`, `boundary-observation` |

## 5. 観測

| 項目 | 内容 |
|---|---|
| 判定シグナル | 失敗時の見方がない、ログ位置が不明、メトリクスやヘルスチェックが弱い |
| 根拠として拾うもの | ログ出力箇所、監視設定、ヘルスチェック、障害手順書 |
| 今すぐ解消すること | 最初に見るログ/メトリクスを1つ決める、どこを確認するかを明記する |
| いま解消しないこと | 監視基盤の新設、広範な observability 改修 |
| エスカレーション先 | `operability-observation`, `observation-minimum-set` |

## 判定メモ

- 2領域以上で source of truth 不明なら `blocked` を優先する
- 要件・境界・検証のどれかが空欄なら `ready` にしない
- 解消可能でも、実装修正に踏み込むなら preflight の範囲外とみなす
- 推奨スキルが4つ以上必要なら、まず一番強いボトルネックを1つに絞る
