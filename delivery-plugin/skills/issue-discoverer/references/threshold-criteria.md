# Threshold Criteria（閾値基準）

カテゴリ別にIssue候補として採用する基準を定義する。
ノイズを減らし、実際にアクションが必要な問題のみを候補化する。

## カテゴリ別基準

### security（セキュリティ脆弱性）

**閾値: なし（常に候補化）**

セキュリティ問題は影響度に関わらず必ず候補化する。
ただし公開Issueには記載せず、Security Advisory経路で報告する。

**例:**
- SQLインジェクション可能な入力パス
- 認証バイパスの可能性
- 機密情報（APIキー、パスワード）のハードコード
- 暗号化なしの機密データ送信
- パストラバーサル脆弱性

### bug（潜在バグ）

**閾値: 実際の機能影響があるもののみ**

以下の条件を**すべて**満たす場合に候補化する:
1. 実行時にユーザーまたはシステムに影響を与える
2. 特定の再現パスまたは条件を説明できる
3. 理論上の可能性ではなく、コードパスから到達可能

**候補化する例:**
- 未処理の例外がクラッシュを引き起こすパス
- 境界値（null, 空配列, 最大値）での予期しない動作
- リソースリーク（未クローズのファイルハンドル、DB接続）
- レースコンディションが発生しうる共有状態アクセス

**候補化しない例:**
- 到達不可能なコードパスの理論的バグ
- フレームワークが保証する範囲内のエラー
- 実害のない型の不一致（暗黙変換が安全に機能）

### missing_tests（テスト不足）

**閾値: クリティカルパスのみ**

以下のいずれかに該当するテスト不足のみ候補化する:
1. **決済・課金処理**: 金銭に直結するロジック
2. **認証・認可**: アクセス制御のロジック
3. **データ永続化**: データ損失に繋がりうる処理
4. **外部API連携**: 障害時のフォールバック処理

**候補化する例:**
- 決済金額計算ロジックにユニットテストがない
- 認証ミドルウェアのエラーケースが未テスト
- データベースマイグレーションのロールバックが未テスト

**候補化しない例:**
- UIコンポーネントの表示テスト不足
- ユーティリティ関数のエッジケーステスト
- 設定ファイルのバリデーションテスト

### tech_debt（技術的負債）

**閾値: 中程度以上（`batch_tech_debt: true` 時はバッチ化）**

個別Issue化する場合の基準:
1. 将来の開発速度に明確に影響する
2. 修正範囲が特定できる（漠然とした「リファクタリング必要」は除外）
3. 非推奨APIの使用で、削除予定が公表されている

**`batch_tech_debt: true` の場合:**
上記の個別基準でフィルタリングした後、残った候補を1つのIssueにまとめる。
バッチモードでも閾値は適用される（全候補を無条件にまとめるわけではない）。
Issue タイトルは `[Tech Debt] <scope> のリファクタリング候補` とし、
body にチェックリスト形式で列挙する。

**候補化する例:**
- 削除予定の非推奨API使用（マイグレーションガイドあり）
- TODO/FIXME コメントに具体的な改善内容が記述されている
- 3箇所以上の明確なコード重複

**候補化しない例:**
- 「もっと良い書き方がある」レベルの改善
- 命名規約の不統一（style カテゴリ）
- 個人の好みに依存するコードスタイル

### style（コードスタイル）

**閾値: 常に除外（候補化しない）**

コードスタイル・命名規約・フォーマットの問題はIssue候補にしない。
これらはリンター/フォーマッターで対応すべきものであり、
手動のIssue管理の対象外とする。
